# Python File Key

This key explains what each `.py` file in the Stage-1 antiviral pipeline does, in plain English.

## Top-level file

### `run_pipeline.py`

**Role:** Main launcher.

This is the file you run from the command line:

```bash
python run_pipeline.py --config example_project/pipeline_config.yaml
```

It imports the command-line interface from `av_pipeline/cli.py` and starts the pipeline.

---

# `av_pipeline/` package

## `av_pipeline/__init__.py`

**Role:** Package identity file.

This marks `av_pipeline/` as a Python package and stores the package version.

You usually do not edit this file.

---

## `av_pipeline/cli.py`

**Role:** Command-line interface.

This file reads command-line arguments such as:

```bash
--config
--init-only
--strict
```

Then it loads the config file and either creates project folders or runs the full pipeline.

Think of this as the “front door” of the program.

---

## `av_pipeline/config.py`

**Role:** Config loader.

This file reads `pipeline_config.yaml` and checks that the required fields exist.

It makes sure the pipeline knows:

- where the run folder is,
- where the input FASTA file is,
- where expected output files are located.

If the config is missing something important, this file raises an error early.

---

## `av_pipeline/utils.py`

**Role:** Shared helper functions.

This file contains small utility functions used everywhere else.

Examples:

- create folders,
- safely read CSV files,
- check whether an external command exists,
- run command-line tools,
- write JSON files,
- convert numeric scores into confidence flags.

Important function:

```python
confidence_from_score()
```

This turns numeric scores into:

```text
HIGH / MODERATE / LOW / FAIL
```

---

## `av_pipeline/setup_run.py`

**Role:** Project folder setup and external-tool marker creation.

This file creates the standard pipeline folder layout:

```text
input/
annotation/
homologs/
msa/
structure/
conservation/
psn/
pockets/
receptor/
ligands/
docking/
reports/
external_markers/
```

It also writes marker files like:

```text
### EXTERNAL TOOL REQUIRED HERE: InterProScan ###
### EXTERNAL TOOL REQUIRED HERE: CDD / CD-Search ###
### EXTERNAL TOOL REQUIRED HERE: GNINA ###
```

These marker files tell you where external program outputs should be placed.

---

## `av_pipeline/fasta.py`

**Role:** Input FASTA validation.

This file checks that the starting input is a valid protein FASTA file.

It verifies:

- the file exists,
- the file contains at least one sequence,
- the sequence looks like amino acids,
- sequence lengths can be summarized.

It also creates short sequence hashes so runs can be tracked.

Output example:

```text
number of records
record IDs
sequence lengths
sequence hashes
```

---

## `av_pipeline/external_tools.py`

**Role:** Optional wrappers for external tools.

This file contains optional code for running some command-line tools automatically if they are installed and enabled in the config.

Currently it includes wrappers/placeholders for:

- InterProScan,
- Pfam/HMMER.

Most external tools are still meant to be run separately at this stage, then their outputs are inserted into the expected folders.

This file is where future automation can be expanded.

---

## `av_pipeline/annotation.py`

**Role:** InterPro/CDD/Pfam parsing and annotation scoring.

This file handles the early protein identity/function step.

It parses:

- InterProScan TSV output,
- CDD/CD-Search CSV output,
- Pfam/HMMER `tblout` output.

Then it normalizes all hits into one table and scores whether the annotation is strong.

It tries to identify target-class buckets such as:

```text
protease
polymerase
helicase/ATPase
methyltransferase
nuclease
entry/fusion
structural
```

Main purpose:

> Decide what the viral protein probably is and how confident we are.

---

## `av_pipeline/homologs.py`

**Role:** VOGDB and homolog-sequence intake.

This file parses viral homolog evidence.

It reads:

```text
homologs/vogdb_hits.csv
homologs/homologs.fasta
```

It checks whether VOGDB/VFAM/VFOLD-style data exists and whether there are enough homolog sequences to support conservation analysis.

Main purpose:

> Decide whether the protein has useful viral evolutionary context.

---

## `av_pipeline/msa.py`

**Role:** MSA conservation analysis.

This file reads MAFFT and MUSCLE alignment files.

Expected files:

```text
msa/mafft_alignment.fasta
msa/muscle_alignment.fasta
```

It calculates simple per-column conservation scores:

- consensus residue,
- gap fraction,
- conservation fraction,
- entropy.

It also compares MAFFT vs MUSCLE to see whether the alignments agree.

Main purpose:

> Turn a multiple sequence alignment into basic conservation evidence.

---

## `av_pipeline/structure_quality.py`

**Role:** Structure-quality parsing.

This file evaluates whether the protein structure is trustworthy enough for pocket analysis and docking.

It can parse:

- AlphaFold/ColabFold-style pLDDT from the PDB B-factor column,
- optional PAE JSON summary,
- optional MolProbity CSV,
- optional QMEAN/QMEANDisCo CSV.

Main purpose:

> Decide whether the structure, especially the pocket region, is reliable enough to use.

External tools involved:

```text
AlphaFold / ColabFold
MolProbity
QMEAN / QMEANDisCo
```

---

## `av_pipeline/conservation.py`

**Role:** ConSurf/custom conservation parsing.

This file reads conservation scores mapped onto the structure.

Expected file:

```text
conservation/consurf_scores.csv
```

Expected columns:

```text
chain,residue_number,residue_name,consurf_grade,conservation_score,exposure
```

It summarizes conservation strength and identifies highly conserved residues.

Main purpose:

> Identify conserved residues that may make a pocket biologically meaningful.

External tool involved:

```text
ConSurf
```

---

## `av_pipeline/psn.py`

**Role:** Protein structure network analysis.

This file reads residue-residue interaction networks.

Expected file:

```text
psn/ring_edges.csv
```

Expected columns:

```text
res1_chain,res1_number,res1_name,res2_chain,res2_number,res2_name,interaction_type,weight
```

It calculates residue centrality using NetworkX if available.

It identifies residues that are highly connected or structurally central.

Main purpose:

> Find residues that may be structurally important inside the protein.

External tools involved:

```text
RING
Cytoscape
RINalyzer
```

---

## `av_pipeline/pockets.py`

**Role:** Pocket parsing and prioritization.

This file parses pocket-prediction outputs from:

- fpocket,
- P2Rank/PrankWeb,
- DoGSiteScorer.

Then it scores pockets based on:

- druggability score,
- pocket score,
- volume,
- independent confirmation,
- overlap with conserved residues,
- overlap with PSN-central residues,
- overlap with functional residues.

Main purpose:

> Rank candidate binding pockets and choose the best pockets for docking.

External tools involved:

```text
fpocket
P2Rank / PrankWeb
DoGSiteScorer
```

---

## `av_pipeline/receptor_prep.py`

**Role:** Receptor-preparation status scoring.

This file does not automatically repair the receptor yet. Instead, it tracks whether receptor-prep steps were completed.

Expected file:

```text
receptor/receptor_prep_status.yaml
```

It checks whether these were marked complete:

- PDBFixer receptor repair,
- ChimeraX manual inspection/cleanup,
- Meeko receptor prep,
- docking box definition.

Main purpose:

> Make sure the receptor is actually ready for docking.

External tools involved:

```text
PDBFixer
ChimeraX
Meeko receptor prep
```

---

## `av_pipeline/ligands.py`

**Role:** Ligand intake, RDKit filtering, and clean SDF generation.

This file reads candidate ligands from:

```text
ligands/raw/ligand_candidates.csv
```

Expected columns:

```text
ligand_id,name,smiles,source,target_class,evidence_type,evidence_notes,known_activity_value,known_activity_units
```

Sources may include:

```text
ChEMBL
BindingDB
PubChem Structure Search
ZINC-22
```

Then RDKit checks each ligand for:

- valid SMILES,
- duplicate structures,
- molecular weight,
- LogP,
- hydrogen bond donors/acceptors,
- TPSA,
- rotatable bonds,
- Lipinski-like properties,
- Veber-like properties,
- PAINS alerts.

It also creates a starter 3D SDF file:

```text
ligands/clean/clean_ligands.sdf
```

Main purpose:

> Convert raw ligand candidates into cleaner, scored docking candidates.

External/source tools involved:

```text
ChEMBL
BindingDB
PubChem Structure Search
ZINC-22
RDKit
Open Babel
Meeko ligand prep
```

---

## `av_pipeline/docking.py`

**Role:** GNINA docking result parsing and scoring.

This file reads summarized GNINA docking results.

Expected file:

```text
docking/gnina_scores.csv
```

Expected columns:

```text
ligand_id,name,pocket_id,cnn_score,cnn_affinity,affinity,pose_file,contacts,notes
```

It scores docking results using:

- GNINA CNN score,
- docking affinity,
- contact annotations,
- pose file availability,
- pocket ID availability.

Main purpose:

> Score whether a docked ligand pose looks computationally plausible.

External tool involved:

```text
GNINA
```

---

## `av_pipeline/scoring.py`

**Role:** Final candidate scoring.

This file combines all evidence layers into one final candidate score.

It combines:

- annotation score,
- homolog/VOGDB score,
- structure-quality score,
- conservation score,
- PSN score,
- receptor-prep score,
- pocket score,
- ligand score,
- docking score.

It creates final candidate rows like:

```text
protein → pocket → ligand → overall score → confidence flag
```

Main purpose:

> Decide which protein–pocket–ligand pairs are worth sending to biophysics simulation.

---

## `av_pipeline/report.py`

**Role:** Final report and manifest generation.

This file writes the final outputs into the `reports/` folder.

Main outputs:

```text
reports/final_handoff_report.md
reports/biophysics_handoff_manifest.json
reports/candidate_table.csv
```

The Markdown report is human-readable.

The JSON manifest is machine-readable and useful for handing off to a later simulation pipeline.

Main purpose:

> Turn all parsed/scored evidence into a clean handoff package.

---

## `av_pipeline/pipeline.py`

**Role:** Master orchestrator.

This is the core file that runs the full workflow in order.

It calls all other modules:

1. FASTA validation
2. optional external-tool wrappers
3. annotation parsing
4. homolog/VOGDB parsing
5. MSA scoring
6. structure-quality scoring
7. conservation parsing
8. PSN parsing
9. pocket prioritization
10. receptor-prep status check
11. ligand filtering
12. docking parsing
13. final candidate scoring
14. report generation

Main purpose:

> Coordinate the entire Stage-1 pipeline from input files to final handoff report.

---

# Most important files to understand first

If you are new to coding, understand these in this order:

1. `README.md` — project overview.
2. `pipeline_config.yaml` — tells the code where files are.
3. `run_pipeline.py` — command you run.
4. `pipeline.py` — master flow.
5. `annotation.py` — InterPro/CDD/Pfam logic.
6. `pockets.py` — pocket scoring.
7. `ligands.py` — ligand filtering.
8. `scoring.py` — final score.
9. `report.py` — final output.

# Files your partner will edit most

Most likely:

```text
pipeline_config.yaml
av_pipeline/annotation.py
av_pipeline/pockets.py
av_pipeline/ligands.py
av_pipeline/scoring.py
av_pipeline/report.py
```

# Files you probably should not edit at first

```text
av_pipeline/__init__.py
av_pipeline/utils.py
av_pipeline/cli.py
```

These are support files and should usually stay stable.
